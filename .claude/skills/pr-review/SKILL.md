# PR前の理解度チェック

## このスキルの目的

PRを作成する前に、変更内容に対する理解度をユーザーに確認する。
「何をやったか分からないままマージされる」ことを防ぐ最終ゲート。

## ルール

### チェック手順

1. **Claudeセルフチェック** — `references/checklist.md` の「Claudeセルフチェック」を自動実行。問題があれば修正してからユーザーに提示する
2. **変更ファイル一覧** — `git diff main --name-only` で変更ファイルを一覧化
3. **変更の要約** — 各変更の概要を初学者向けに説明する
4. **理解度チェック** — `references/checklist.md` の「必須チェック」に沿ってユーザーに確認
5. **記録漏れの検出** — 以下をチェックし、漏れがあれば提案する
   - 設計判断があったのにADRがない → `adr` スキルの利用を提案
   - 新技術を使ったのに学習ノートがない → `learning-note` スキルの利用を提案
6. **TODOコメントの提案** — 理解が不十分な箇所があれば `// TODO: 理解する - [概要]` コメントの追加を提案

### チェック完了後

- ユーザーがチェックリストを確認したら、PR作成に進む
- `.github/PULL_REQUEST_TEMPLATE.md` のテンプレートに沿ってPR本文を作成する

### チェックの提示フォーマット

```
## PR前の理解度チェック

### 変更ファイル（N件）
- `path/to/file.rb` — （変更の概要を1行で）
- `path/to/file.tsx` — （変更の概要を1行で）

### 確認させてください
1. このPRで変更したコードの動作を、自分の言葉で説明できますか？
2. 新しく導入したライブラリ・パターンはありますか？その役割を理解していますか？
3. このコードが壊れやすいポイントはどこですか？

### 記録チェック
- ✅ / ❌ 設計判断のADR
- ✅ / ❌ 新技術の学習ノート
```
