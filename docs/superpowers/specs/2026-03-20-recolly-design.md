# Recolly（レコリー） — プロダクト設計仕様書

## 1. 概要

### 1.1 プロダクト名
**Recolly**（レコリー）— Record + Collection

### 1.2 コンセプト
物語性のあるメディア（アニメ、映画、ドラマ、本、漫画、ゲーム）をジャンル横断で記録・分析・共有するWebアプリケーション。

### 1.3 差別化
- **コア**: 全ジャンルをオールインワンで管理できること自体が価値（既存サービスはジャンルごとに分断）
- **付加価値**: ユーザーの記録データに基づいたクロスメディアレコメンド（UIでは「おすすめ」として自然に表示。「AI」という言葉はUIに出さない）

### 1.4 対象ユーザー
ユーザー登録ありの公開サービス。アニメ・映画・本・ゲームなど複数ジャンルのコンテンツを楽しむ人がターゲット。

### 1.5 収益モデル
完全無料。ポートフォリオ兼実用プロダクトとして開発。

### 4.2 リレーション
- User → 複数の UserProvider（1対多）
- User → 複数の Record（1対多）

- Discussion → 複数の Comment（1対多）

- **media_type**: anime, movie, drama, book, manga, game
- **status**: watching, completed, on_hold, dropped, plan_to_watch

## 5. 認証

### 5.1 認証方式
| 方式 | 説明 |
|------|------|
| メール + パスワード | 通常のメール登録 |
| Google OAuth | Googleアカウントでログイン |
| X (Twitter) OAuth | Xアカウントでログイン |

### 5.2 設計方針
- OAuth情報はUserProvidersテーブルで管理（1ユーザー = 複数プロバイダ連携可能）
- 既存アカウントへのOAuthプロバイダ追加は設定画面から行う（ログイン済み状態で連携）

### 5.3 同一メールアドレス衝突時のルール
- **メール登録済み → 同一メールのOAuthでログイン試行**: エラーとし「このメールアドレスは既に登録されています。メールアドレスでログインしてください」と案内
- **OAuth登録済み → 同一メールの別OAuthでログイン試行**: エラーとし「このメールアドレスは既にGoogleで登録されています」と案内
- **OAuth登録済み → 同一メールでメール登録試行**: 同上、既存のログイン方法を案内
- **OAuthプロバイダがメールアドレスを返さない場合**（Xはメール取得がオプショナル）: メールアドレスなしでアカウント作成を許可し、初回ログイン後にメールアドレスの設定を促す。メール未設定ユーザーはメール必須の機能（パスワードリセット等）が使えない

## 8. システム構成

```
ユーザーのブラウザ（React + TypeScript）
  ↕ API通信（JSON）
バックエンド（Ruby on Rails API）
  ├── 認証（Google / X / メール）
  ├── 記録管理（CRUD、進捗、評価）
  └── 掲示板（スレッド、コメント）
  ↕
  ├── PostgreSQL（AWS RDS）— 全データ保存
  ├── 外部API（TMDB, AniList, Google Books, IGDB）— 作品データ取得
  └── レコメンドエンジン — 好み分析、おすすめ生成

AWS インフラ:
  EC2（Railsサーバー）
  RDS（PostgreSQL）
  S3（画像ストレージ）
  CloudFront（CDN配信）
```

## 11. 非機能要件

### 11.1 テスト
- TDD（テスト駆動開発）を全機能に適用
- バックエンド: RSpec
- フロントエンド: Jest + React Testing Library

### 11.2 コード品質
- バックエンド: RuboCop（Rubyリンター）
- フロントエンド: ESLint + Prettier
- コミット前にlint + テスト自動実行（hooks）
- 未使用コード（デッドコード）検出をlintルールに含める

### 11.3 セキュリティ
- パスワードのハッシュ化（bcrypt）
- CSRF対策
- SQLインジェクション対策（Rails標準のパラメータバインディング）
- XSS対策（React標準のエスケープ + 追加対策）
- APIキー・シークレットは必ず環境変数で管理（ハードコード禁止）
- 全APIエンドポイントに認証チェックを必須とする
- 入力バリデーションをコントローラーレベルで必ず実施
- 依存パッケージの脆弱性チェック（bundle audit / npm audit）

### 11.4 AI駆動開発における品質防止策（CLAUDE.mdに反映）
AI（Claude）がコードを生成する際に守るべきルール。バイブコーディングの既知の失敗パターンを防止するためのガードレール。

| ルール | 理由 |
|--------|------|
| 色・フォントサイズ等のデザイントークンはCSS変数のみ使用。ハードコード禁止 | AI生成コードは値をハードコードしがちで、後から一括変更できなくなる |
| マジックナンバー禁止。定数として定義する | AI生成コードは数値リテラルをそのまま使いがち |
| 新規ページ作成時は既存の共通コンポーネントを必ず使用。新規スタイル直書き禁止 | ページごとにUIがブレる問題の防止 |
| 未使用のimport・変数・関数を残さない | デッドコードの蓄積防止 |
| 1ファイル200行以内を目安。超える場合は分割を検討 | AI生成コードは1ファイルに詰め込みがち |
| コードの意図がわかるコメントを書く（何をしているかではなく、なぜそうしているか） | 自分が書いてないコードの理解を助ける |
| 外部APIのレスポンスは必ず型定義し、バリデーションする | AI生成コードは外部データを信頼しがち |
| DBクエリにはN+1問題対策（includes/eager_load）を必ず考慮 | AI生成コードはパフォーマンスを無視しがち |
| フェーズ完了ごとにコードレビューを実施 | 技術的負債の早期発見 |
