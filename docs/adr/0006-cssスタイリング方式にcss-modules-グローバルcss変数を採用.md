# ADR-0006: CSSスタイリング方式にCSS Modules + グローバルCSS変数を採用

## ステータス
承認済み

## 背景
フェーズ0bでデザインシステム基盤を構築するにあたり、CSSの管理方法を決める必要がある。デザイントークン（色・フォントサイズ・余白等）の定義方法と、各コンポーネントのスタイルのスコープ管理を決定する。

## 選択肢

### A案: CSS Modules + グローバルCSS変数
- **これは何か:** デザイントークンは普通のCSSファイルに `--color-primary: #2c2c2c` のような変数として定義し、各コンポーネントのスタイルは `.module.css` という特殊なファイルに書く。`.module.css` に書いたクラス名は自動的にそのコンポーネント専用になり、他と衝突しない。
- **長所:** 追加ライブラリ不要（Viteが標準サポート）、普通のCSSなので読みやすい、仕様書の「CSS変数のみ使用」ルールにそのまま準拠
- **短所:** 動的スタイル（状態に応じてスタイルを変える）がやや面倒

### B案: Tailwind CSS
- **これは何か:** `className="bg-white p-4 flex"` のように、HTMLのクラス名にスタイルを直接書くユーティリティファーストのCSSフレームワーク。
- **長所:** 人間が高速でUIを書ける、クラスが統一される
- **短所:** 学習コストが高い、クラス名が長く初学者には読みにくい、デザイントークンの管理場所が設定ファイルに分散

### C案: styled-components
- **これは何か:** JavaScriptファイルの中にCSSを書く方式（CSS-in-JS）。`const Button = styled.button` のように書く。
- **長所:** 動的スタイルが得意、コンポーネントとスタイルが一体化
- **短所:** ランタイムコスト（ページ表示時にCSS生成が走る）、追加パッケージが必要、CSSとJSが混在して初学者には複雑

## 決定
**A案: CSS Modules + グローバルCSS変数** を採用する。

## 理由
- CLAUDE.mdに「デザイントークンはCSS変数のみ使用」と明記しており、A案がそのまま準拠できる
- AI駆動開発において、ルールが「`var(--color-xxx)` を使え」と単純明快なほうがAIが逸脱しにくい
- 追加依存ゼロで、壊れるポイントが少ない
- 普通のCSSなので、ブラウザDevToolsでそのまま確認・デバッグできる
- B案（Tailwind）はAIが書く場合、高速開発の利点が薄れ、設定の複雑さだけが残る
- C案（styled-components）はランタイムコストと追加依存が不要な複雑さを持ち込む

## 影響
- 全コンポーネントのスタイルは `.module.css` ファイルに記述する
- デザイントークンは `frontend/src/index.css` の `:root` にCSS変数として一元管理する
- 動的スタイルが必要な場合は、クラス名の切り替え（`className={条件 ? styles.active : styles.default}`）で対応する
