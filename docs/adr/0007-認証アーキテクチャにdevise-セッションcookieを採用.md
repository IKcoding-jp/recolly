# ADR-0007: 認証アーキテクチャにdevise + セッションCookieを採用

## ステータス
承認済み

## 背景
フェーズ1（MVP）でユーザー認証機能を実装するにあたり、以下の技術選定が必要になった。
RecollyはRails APIモード + React（フロントとバックが別サーバー）の構成。
AI駆動開発のため、車輪の再発明を避け既存ライブラリを最大限活用する方針。

将来的にモバイルアプリ対応やAPI公開の可能性があるが、時期は数ヶ月〜1年以上先の見込み。
MVPではWebブラウザのみを対象とする。

## 選択肢

### 1. トークン管理方式

#### A案: セッションCookie
- **これは何か:** Railsが標準で持つ仕組み。ログインするとサーバーがセッション情報を作り、ブラウザにCookieとして渡す。以降のリクエストでブラウザが自動でCookieを送り、サーバーが「この人はログイン済み」と判断する
- **長所:** deviseの標準動作そのまま使える。httpOnlyがデフォルトでセキュリティが高い。カスタム実装不要
- **短所:** APIモードではデフォルト無効（ミドルウェア2行追加で有効化）。モバイルアプリでは使いにくい

#### B案: JWT（JSON Web Token）
- **これは何か:** ユーザーIDや有効期限などの情報を暗号化して1つの文字列にしたもの。サーバーはDBを見なくても、この文字列を復号すれば誰かわかる
- **長所:** モバイルアプリやAPI公開に対応しやすい。スケールしやすい
- **短所:** トークンの無効化が難しい。httpOnly Cookieで送る場合はカスタム実装が必要で、車輪の再発明になる

#### C案: DBトークン
- **これは何か:** ログイン時にランダムな文字列を生成してDBに保存し、照合する方式
- **長所:** シンプルで理解しやすい
- **短所:** 毎回DBへの問い合わせが発生。自前実装が必要

### 2. 認証ライブラリ

#### A案: devise
- **これは何か:** Railsで最も有名な認証ライブラリ。パスワードのハッシュ化、ログイン/ログアウト、パスワードリセット、アカウントロック等の機能が揃っている
- **長所:** 機能が豊富で実績がある。車輪の再発明を避けられる。将来OAuthを追加する際もomniauthableモジュールで拡張可能
- **短所:** 内部の処理が多く、仕組みを完全に理解するのは難しい

#### B案: has_secure_password + 自前実装
- **これは何か:** Rails標準のパスワードハッシュ化機能を使い、認証フローを自分で実装する
- **長所:** 仕組みが透明で理解しやすい。軽量
- **短所:** パスワードリセット等を自前で実装する必要がある。車輪の再発明

#### C案: rodauth-rails
- **これは何か:** モジュラー型の認証ライブラリ。必要な機能だけ選んで追加できる
- **長所:** モジュラー設計で柔軟。セキュリティのデフォルトが堅い
- **短所:** コミュニティが小さく日本語情報がほぼない

### 3. フロントエンドのトークン保存場所

セッションCookieを採用するため、この判断は不要になった。
ブラウザがCookieを自動管理するため、フロントエンドで保存場所を意識する必要がない。

## 決定
1. トークン管理方式: **セッションCookie**
2. 認証ライブラリ: **devise**
3. フロントエンドのトークン保存: **判断不要**（ブラウザが自動管理）

## 理由
- **セッションCookie:** deviseの標準動作をそのまま使える最もシンプルな選択。httpOnlyがデフォルトでXSS攻撃からも安全。モバイル対応は数ヶ月〜1年以上先のため、YAGNI原則に従いJWTは必要になった時点でdevise-jwtを追加する（移行ではなく追加で対応可能）
- **devise:** AI駆動開発で車輪の再発明を避ける方針。パスワードリセット、アカウントロック等が揃っており自前実装のリスクを回避。has_secure_passwordは理解には良いが、既存ライブラリ活用を優先。rodauthは情報の少なさがリスク
- **トークン保存が不要に:** セッションCookieはブラウザが自動管理するため、localStorageやhttpOnly Cookieへの手動保存の判断自体が不要になった。これも「車輪の再発明を避ける」方針の帰結

## 影響
- `backend/Gemfile` に `devise` を追加（devise-jwtは不要）
- Rails APIモードにセッション用ミドルウェアを追加（`ActionDispatch::Cookies`, `ActionDispatch::Session::CookieStore`）
- CORS設定で `credentials: true` を指定（`rack-cors` gem）
- フロントエンドのトークン管理コードは不要
- 将来モバイルアプリ/API公開が必要になった時点で `devise-jwt` を追加し、Web（セッション）とモバイル/API（JWT）を並行運用する
